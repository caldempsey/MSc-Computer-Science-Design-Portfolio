Good evening COMP1026 examiners,

As Matthew is aware I have encountered incompatibility issues with the object tag and Javascript functions I have tried to pass through it: that is the object tag fails to update the content of the interactive window consistently where iframes do not fail thereto. Consequently that website will not validate in xHTML (because of the use of iframes). To that end I have prepared three presets of the same website.

First of these is option A (in directory evolution_primary) which was the originally intended submission for the purposes of grading: featuring Javascript error handling whereby if the user has Javascript enabled the page is outright transformed into a much more fluid feeling and much faster loading page on servers that support Javascript. Unfortunately the www.personal.leeds.ac.uk webpage does NOT support Javascript, as informed by the ISS helpdesk. This issue is serverside, not clientside: you may observe how Javascript calls jQuery functions correctly but fails to pass them, forcing it to reiterate over the same function until called, crashing the browser. So it is impossible for my detection methods to pick up (since they run on Javascript and if Javascript is enabled in browser then it tries to run functions the server will not pass). Otherwise the website appears compatibile with all browsers. Therefore, if you are to browse the first option from personal.leeds.ac.uk I urge you to disable Javascript in your browsers (enabling the sites Accessibility mode feature) or run the website package locally or see the website of Option A running via a different host.

Option B on the other hand (in directory evolution_secondary) does not obtain so many Javascript features (the iframe function has been removed) and has been made thereto fully xHTML authenticated. Changes are as follows: discontinued the use of THE iframe class, so all websites do not load quickly in index.html; changed use of java.js to java_secondary; and removed accessibility function consequently (no longer necessary since no iframes other than those authenticated by Matthew (chat and videos, although please note Matthew authenticated the iFrame swap which takes place in Option A's index.html).

However Option B as well as Option A employs Javascript to some end, so is incompatible with the domain specified for the requirements of the module. For this please try Option C, where all incompatible Javascript features are stripped. The only significant change from Option B is that the mock user-authentication feature has been removed.

Option A's accessibility mode which runs when Javascript is disabled appears to be ideal for text-based browsers (since it is easier to browse the content than having to scroll through the index.
